## Item A2 — Production OAuth2 PKCE auth-flow factory

### Grounding (what exists)
- The generic client-driven auth seam is complete and live:
  - Trait `AuthFlowFactory` + `PendingAuthFlow` + `AuthOutcome` in
    `crates/substrate/daemon-host/src/auth.rs`; registry `PendingAuthFlows` parks/evicts flows.
  - Wire types (in `crates/contracts/daemon-api/src/lib.rs`, **do not touch**): `AuthApi`
    (`auth_providers`/`auth_begin`/`auth_complete`/`auth_cancel`), `AuthFlowKind::{MatrixSso,
    OAuth2Pkce}` (line ~1425), `AuthBeginRequest{ family, params: BTreeMap<String,String>,
    redirect_uri, bind }`, `AuthProviderInfo`, `AuthParamField`, `AuthCompleteResponse`.
  - The node's `AuthApi` impl (`crates/substrate/daemon-host/src/node_api/cred_auth.rs`) already
    writes the returned `AuthOutcome.credential_blob` into the `CredentialStore`, honors the optional
    profile bind, and rebuilds routing — proven end-to-end by the **stub** factory test in
    `tests/daemon-conformance/src/node/routing_auth.rs::interactive_auth_generic_begin_complete_binds_and_lists`.
  - The Matrix reference factory `MatrixAuthFlowFactory` lives in the **adapter** crate
    (`crates/adapters/daemon-matrix/src/auth.rs`), not in `daemon-host` — deliberately, to keep the
    heavy SDK out of the host. It is registered in `bins/daemon/src/main.rs` (~line 2389) into
    `auth_factories: Vec<Arc<dyn daemon_host::AuthFlowFactory>>` (gated on `cfg.matrix.enabled`),
    then threaded through `NodeAssembly.auth_factories`.
- Spec `crates/engine/daemon-core/docs/daemon-interactive-auth-spec.md` §7 + roadmap item **A2**
  describes exactly this: a generic `OAuthAuthFlowFactory` (PKCE + `state`), `params` carrying
  `{ issuer/authorization_endpoint, token_endpoint, client_id, scopes }`, `begin` mints verifier +
  challenge + state + URL, `complete` validates `state`, exchanges `code`+`verifier` at the token
  endpoint, serializes the token set as the blob. "No new wire surface is needed."
- **No existing provider currently references PKCE config** (grep of `crates/providers`,
  `daemon-credentials`, cloud catalog found only doc/comment mentions). So the PKCE config shape is
  greenfield — we own it. The only production PKCE consumer today is the (test-only) stub.
- Dependencies available: `oauth2 = 5.0.0` is already in `Cargo.lock` (transitive via matrix-sdk);
  workspace already pins `reqwest 0.12` (json/rustls), `sha2 0.10`, `base64 0.22`. `url` is in the
  lock (transitive) but not a workspace dep.

### Design decision — new crate `crates/adapters/daemon-oauth`
Mirror the Matrix precedent: the factory lives **outside `daemon-host`** so the host stays free of an
HTTP client. New crate `daemon-oauth` depends on `daemon-host` (for `AuthFlowFactory`/
`PendingAuthFlow`/`AuthOutcome`), `daemon-api` (`AuthProviderInfo`/`AuthFlowKind`/`AuthParamField`/
`ApiError`), `daemon-protocol` (`TransportId`), plus `reqwest`, `sha2`, `base64`, `serde`,
`serde_json`, `async-trait`, `thiserror`, and a URL builder.

**Crypto/HTTP approach (decision):** implement PKCE by hand with workspace deps rather than pulling
the `oauth2` crate as a *direct* dep. Rationale: (1) PKCE is small and the codebase already hand-rolls
this shape (`fresh_flow_id` in `auth.rs` does manual SHA-256); (2) avoids a `reqwest 0.12` vs
`oauth2-reqwest`(reqwest 0.13) split and any `cargo deny`/`machete` churn; (3) full control over the
`state`/`redirect_uri` echo checks the spec mandates. *Alternative kept in reserve:* use `oauth2 = 5`
with a custom async-http closure over the workspace `reqwest 0.12` — cleaner enforcement but a new
direct dep. **Recommend hand-rolled; flag the choice for review.**

**Family / config decision:** register a single generic family `"oauth2"` that is **params-driven**
(no config table required): the client supplies endpoints + client_id + scopes in
`AuthBeginRequest.params`. This makes it live immediately (matches the ui-gaps "Custom endpoint /
OAuth sheet" vision) with zero new config surface. *Optional follow-up (flagged, not in core):* a
`[[oauth.provider]]` config table of named providers (family = provider name, endpoints/client_id
baked in) for nicer UX. **Recommend params-driven for this fix.**

          `reqwest`;
       3. blob = the JSON token response (access/refresh/expiry/id_token) as the `credential_blob`;
       4. `credential_ref` = params `account_label` (or derive from `id_token`/`sub` when present,
          else a stable hash of client_id+subject) — **flagged open question below**;
       5. `transport_instance = TransportId::new(format!("oauth2/{label}"))`,

## Item B3 — routing_builder hot-reload hole

### Grounding — the scope's description is partly stale; here is the real hole
- **Chat-pin unbind already hot-reloads.** `RoutingUnbindChat`/`RoutingBindChat`/`RoutingSet`
  (`crates/substrate/daemon-host/src/node_api/control.rs` ~620–656) **all** call
  `load_routing_pins().await` → `rebuild_routing()`. The conformance test
  `routing_auth.rs::routing_pin_resolves_to_bound_session` already asserts unbind clears the pin and
  `routing_get` returns `None`. **So "unbinding a chat does not trigger a rebuild" is FALSE in current
  code** (scope contradiction — see final summary).
- **The real hole (matches user-stories EIO-3/EIO-7):** the `routing_builder` rebuild hook is
  **never installed in production**. `bins/daemon/src/main.rs` builds a *static* registry
  (`build_routing_registry(&cfg.routing)`) and passes it as `routing`; it never calls
  `with_routing_builder`. In `crates/node/daemon-node/src/assembly/mod.rs::install_routing`
  (~line 790) the account→profile baseline (`instance_profiles`) is derived **once at assembly**
  via `RoutingRegistry::bind_instances_from_profiles(&profile_specs)` and installed with
  `with_routing` (a boot-time snapshot).
  - After boot, editing a profile's `bound_accounts` (via `profile_update`, or the account bind in
    `auth_complete`) DOES call `NodeApiImpl::rebuild_routing()` (`node_api/profile.rs:55`,
    `node_api/cred_auth.rs:113`) — but with `routing_builder = None`, `rebuild_routing`
    (`node_api/routing.rs:23`) just re-clones the **stale** `routing_base`. So a newly-bound or
    unbound account never changes the live `instance_profiles` until restart.
- `bind_instances_from_profiles` uses `or_insert_with` → explicit config bindings win over
  profile-declared ones (operator override preserved). `RoutingRegistry: Clone`. The
  `with_routing_builder(builder)` seam exists and calls `rebuild_routing()` immediately (seeds live).
- The conformance harness (`tests/daemon-conformance/src/node/harness.rs::assemble_node` →
  `daemon_node::assemble` → `install_routing`) exercises the exact production path, so a fix here is
  provable in-crate.

### Design decision — install a profile-driven rebuild hook
In `install_routing`, when a **profile store is present**, install a `RoutingBuilder` closure
(instead of a one-shot static registry) that recomputes the base from the *live* profile store on
every rebuild. When no profile store is present (many low-level harness assemblies), keep today's
static behavior.

### File-by-file plan
1. `crates/node/daemon-node/src/assembly/mod.rs` — rewrite `install_routing`:

       match a.profiles.clone() {

## Item F4 — Operator-steer conformance proof

### Grounding — verify the scope's premise against code
- `SessionApi::submit(session, command)` (`crates/substrate/daemon-host/src/node_api/session.rs:8`)
  **claims the session for the Live lifecycle** (`self.claim(&session, Lifecycle::Live)`), which
  *rejects an id already claimed by the Durable lifecycle* (`node_api/internals.rs::claim`,
  `owners: DashMap<SessionId, Lifecycle>`).
- Delegated children are created by the fleet **job worker** (`crates/node/daemon-node/src/fleet/
  job_worker.rs`) directly via `store.create_session` + `enqueue_wake` — they are **never claimed via
  the wire `assign`**, so they are **absent from `owners`**. Therefore an operator `submit(child, …)`
  succeeds at the claim step (vacant → Live) and runs.
- **Important nuance (verified):** the live session builder in `daemon-node/src/assembly/mod.rs`
  (~line 630) always constructs a **fresh** engine (`ctx.resolve_effective(spec, overlay).fresh(id)`
  / `fallback.fresh(id)`). It does **not** hydrate from the durable child snapshot. So an operator
  `submit(child, Steer/StartTurn)` opens a **new blank live engine over the child's id** and runs the
  operator's command on it — the child "receives/executes" the command, but this is a live incarnation
  distinct from the child's durable delegated conversation (a latent lifecycle fork; harmless when the
  child has already reached a terminal durable state). **This is the key F4 open question.**
- `Assign(session)` (`node_api/control.rs:390`) claims **Durable** and wakes; the durable activation
  path DOES rehydrate from the store snapshot. `Assign` carries **no** steer text.
- Delegation harness precedents: `tests/daemon-conformance/src/node/detached_delegation.rs`
  (assemble node, drive children, `store.children_of(parent)`, poll for `TurnFinished`) and
  `f3f4_ownership.rs` (`node.assign(parent)` → default node delegates once → child Engine unit in the
  tree). `gate_providers()` (harness): default=completing, `orchestrator`=delegating once,
  `child`=completing. `AgentCommand::{StartTurn, Steer}` exist; `Steer` "opens a steer turn when the
  engine is idle". The live `submit` records the inbound command on the merged log (so the operator's
  text is observable via `poll`/snapshot even with a fixed-reply provider).

### Design decision
Prove the two provable statements the scope asks for, honestly:
1. **Operator addresses a delegated child by id and it executes** — via `submit(child, StartTurn/
   Steer)`, asserting a fresh `TurnFinished` on the child and the operator's text in the child's live
   transcript. (Document that this is a fresh live incarnation, not a durable resume.)
2. **Parked-durable child wakes via `Assign`** — a durable child parked (suspended) resumes when the
   operator calls `Assign(child)`.

  3. Operator: `node.submit(child.clone(), AgentCommand::StartTurn{ input: UserMsg::new(
     "OPERATOR-STEER-PING"), request_id })` (also cover `Steer{ text }`).
  4. `poll(child)` to a `TurnFinished`; assert (a) a turn ran, (b) the child's transcript
     (`Snapshot`/merged log) contains `OPERATOR-STEER-PING` — proving receipt/execution.
- Test `operator_assign_wakes_a_parked_durable_child` (the parked variant; harness supports it):
  - Seed a **parked durable child** directly (the `detached_delegation.rs::
    detached_notice_reaches_a_parked_durable_parent` pattern: `store.create_session` with a
    `Snapshot` carrying `waiting_for` + a `PendingApproval`, id shaped as `{parent}/c1`), let the
    recovery scanner park it, then `node.assign(child)` and assert it wakes/advances (status leaves
    `Suspended` toward resolution, or a wake is observed). If tying "steer" to the parked child proves
    impossible via a pure wire op (no input-carrying durable op — see open question), keep this case
    as the "operator resumes a parked child" proof and document the limitation inline.

### Risks / notes (F4)
- The primary test asserts the *observable* (child ran + operator text present). Whether F4's intended
  semantics require a **durable resume** (fold operator input into the child's existing delegated
  conversation) vs. the current **fresh-live-engine** behavior is the central open question — it may
  reveal a deeper node gap beyond a conformance test. **Flagged.**
- `submit`-Live over a not-yet-terminal durable child could genuinely fork the session; the test uses
  a child that has reached a terminal durable state to keep the proof clean and non-flaky.
- Discover the child id via `store.children_of(&parent)` (robust) rather than assuming `{parent}/c1`.
