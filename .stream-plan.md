- DURABLE sessions: dormant between turns; every activation rehydrates `session_allow` from the
  persisted snapshot → persist-only revoke ACCEPTED and implemented (takes effect at the next
  activation; documented on the op). The edit rides a new store compare-and-swap
  (`swap_snapshot_if_dormant`) so an Active incarnation (whose checkpoint would overwrite the
  edit) or a concurrent writer REFUSES instead of losing the revoke.
- LIVE-resident sessions: verified the resident engine starts `fresh()` and NEVER reads or
  persists the durable snapshot — its allow-list is memory-only and dies at session close. So
  instead of an engine-reachable mutation (new §17 command surface + foreign-codec fallout),
  both ops REFUSE on a live-resident session with a clear Unsupported error. Recorded follow-up:
  extend reach into the resident engine if live-session fingerprint management becomes a need.
- Acceptance bar met honestly: conformance (`fingerprint_manage.rs`) drives permanent-allow →
  auto-approve of the identical command → list → revoke → the SAME command RE-PROMPTS (a fresh
  park carrying the same fingerprint), all through the wire ops, no backdoors.

## 4. D4 — approval fingerprint list/revoke

### 4a. Storage reality
Per-session allow-list is `Snapshot.session_allow_fingerprints: Vec<CommandFingerprint>`
(core/src/snapshot.rs:80–90). It is added by `Effect::RememberApproval` (turn.rs:137–144) →
`Engine::remember_session_allow` (engine.rs:225–237) and read via `TurnCx.session_allow` seeded from the
snapshot each round (turn.rs:61–68; short-circuit at ask_host ~236–239). `CommandFingerprint` is an opaque
SHA-256 hex (core/src/exec/mod.rs:198–251). There is NO separate table and NO wire op today.

### 4b. New ops (api)
- `ApiRequest::FingerprintList { session: SessionId }` → `ApiResponse::Fingerprints(Vec<RememberedFingerprint>)`.
- `ApiRequest::FingerprintRevoke { session: SessionId, fingerprint: String }` → ok/err.
- New DTO `RememberedFingerprint { fingerprint: String, #[serde(default)] label: Option<String> }`
  (api/src/lib.rs). `label` is `None` for now (storage keeps only the hash) but future-proofs the shape so a
  human-readable command surface can be added without a wire bump. (See Q4 — bare `Vec<String>` is the
  alternative.)
- ControlApi defaults: `async fn fingerprint_list(&self, _s) -> Vec<RememberedFingerprint> { vec![] }`,
  `async fn fingerprint_revoke(&self, _s, _fp) -> Result<(),ApiError> { Err(Unsupported(...)) }`.
- `dispatch.rs` — route in `serve_session` (they are session-scoped, alongside ApprovalDecide).
- CDDL: `remembered-fingerprint = { "fingerprint": tstr, ? "label": (tstr / null) }`,
  `request-fingerprint-list = { "FingerprintList": { "session": session-id } }`,
  `request-fingerprint-revoke = { "FingerprintRevoke": { "session": session-id, "fingerprint": tstr } }`,
  `response-fingerprints = { "Fingerprints": [* remembered-fingerprint] }`; add all three to the unions.

### 4c. Host impl + engine reach (see Q5 — the main D4 risk)
- authz.rs: `FingerprintList` → `SessionWrite`? No — `SessionRead`; `FingerprintRevoke` → `SessionWrite`.
  Add to the `SessionRead`/`SessionWrite` arms (~L80–88, next to ApprovalDecide).
- ownership_matrix.rs (verified: `ApprovalDecide {..} => OwnerGated(Forbidden)` at L267, sample at L544):
  both fingerprint ops carry `session` → **OwnerGated**. `FingerprintList` (read) → `OwnerGated(EmptyOrAbsent)`
  (like `ApprovalsPending`, L264); `FingerprintRevoke` (write) → `OwnerGated(Forbidden)`. Add both to the
  exhaustive `match` AND add samples targeting alice's session `s` in `owner_gated_samples` (L430+).
- Impl strategy:
  - **Read (`fingerprint_list`)**: read the durable `Snapshot` for the session via the store (pattern like
    `get_approval_policy`/snapshot read in control.rs), map `session_allow_fingerprints` →
    `RememberedFingerprint { fingerprint: fp.as_str().into(), label: None }`.
  - **Revoke (`fingerprint_revoke`)**: read-modify-write the durable snapshot, dropping the matching
    fingerprint, then persist. For a LIVE session whose engine holds a resident snapshot, the mutation must
    reach the running engine (its `TurnCx.session_allow` reseeds from the snapshot each round, so a persisted
    change is picked up on the next round IF the engine reloads from the store — verify). If the live engine
    keeps an in-memory snapshot that is not reloaded per round, add an engine command to drop the fingerprint
    (new `AgentCommand`/live-handle method) and persist. **This is the open design point (Q5).**

## 7. F2 — notice correlation (see Q8 — biggest design decision)

### 7a. Problem
`node/src/fleet/notice_worker.rs:45` injects a plain `format!("[subagent {} completed] {}")` as a bare
`UserMsg` (via `NodeApiImpl::inject_session_input`, host/src/node_api.rs:485–528). No child-unit/call_id
provenance travels, so a client cannot chip-link the completion turn back to the delegation card (cards are
keyed by the parent orchestrate tool `call_id`, `ToolCallView.call_id`, proto:441). The parent's detached
spawn `call_id` is NOT persisted on `JobCommand` or `CompletionNotice` today.

### 7b. Recommended design (lowest-churn that satisfies chip-linking)
1. **Persist the spawn call_id.** Add `call_id: Option<String>` to `store::CompletionNotice` (store/src/lib.rs:364–372)
   and to the completion-notice edge write path. In `orch-tool/src/lib.rs` detached spawn (~L459–495), pass
   the current `ToolCall.call_id` (in `cx`) into the notice-binding call. Requires the store
   `bind_completion_notice`/`mark_completed` signatures to carry the call_id (store-internal, no wire change;
   SQLite outbox table gains a nullable `call_id` column — small migration in sqlite.rs + schema.golden.sql).
2. **Carry provenance on the injected turn (wire).** Add an optional field to `proto::UserMsg`
   (proto/src/lib.rs:26–34):
   ```rust
   #[serde(default)] pub notice: Option<CompletionNoticeRef>,
   ```
   with `CompletionNoticeRef { child: UnitId, #[serde(default)] call_id: Option<String> }` (proto). CDDL:
   extend `user-msg` with `? "notice": (completion-notice-ref / null)` and add the rule. `UserMsg` is the
   `StartTurn` payload AND the durable pending-input encoding (proto:51–66) — the notice rides the existing
   live + durable inject seam with no new plumbing.
3. **Notice worker** builds `UserMsg { text, notice: Some(CompletionNoticeRef { child, call_id }) }` and adds
   a host method `inject_session_input_msg(session, UserMsg)` (or make `inject_session_input` take a
   `UserMsg`), so the structured field survives instead of being flattened to text.

Client (daemon-app) consumption is OUT OF SCOPE; the wire now carries what a client needs.

**Alternative (Q8):** instead of `UserMsg.notice`, tag the injected turn via
`TurnTrigger::BackgroundCompletion { source: CompletionSource::Delegation(job) }` (proto:293) — this already
exists, but keys on `JobId`, not the parent `call_id`, so it does NOT directly chip-link to the delegation
card. Recommend the `UserMsg.notice` approach; flag for coordinator.

### 7c. Conformance
- New file `conf/tests/notice_provenance.rs` (the existing `conf/src/node/detached_delegation.rs` covers the
  drain; keep it): detached child completes, assert the parent's injected turn carries
  `notice = Some(CompletionNoticeRef { child, call_id })`. CBOR round-trip of `UserMsg { notice: Some(..) }`.
